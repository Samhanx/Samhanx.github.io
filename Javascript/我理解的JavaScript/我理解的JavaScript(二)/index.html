我理解的JavaScript(二)

发表于 2017-04-27 | 分类于 Javascript ， 我理解的JavaScript

第一篇文章简单介绍了JS引擎，目的还是为了建立起这个概念，方便理解作用域，执行上下文，变量对象和作用域链这一系列核心概念。

作用域

什么是作用域

作用域是我们在学习JS时常常看到的一个词，它往往便随着变量出现在各类文章或者面试题里，实践中也常给新手造成代码报错或者代码输出结果与预期不相符等困扰。那么它到底是一个什么鬼？

这里用自己的理解对它进行描述：作用域是语言标准定义的一套规则，程序根据它来决定如何存储以及访问进而修改变量。

作用域又分静态作用域（又叫词法作用域）和动态作用域，大多数程序设计语言，包括JavaScript，采用的都是静态作用域。

静态作用域在代码的编译阶段进行建立，确定了当前可执行代码对变量的访问权限。具体到JS来说，个人理解它就是ECMAScript定义的一套语言规则，JS引擎根据它在编译代码时确定当前代码存储了哪些变量。

请注意，作用域规则是在JS引擎编译代码时就已经确立好的，通俗一点说，当我们写下代码的时候，就明确了哪一段代码可以访问到声明的哪些变量和函数。

以上是一些绕口的概念性的东西，觉得太抽象不好理解也没什么关系，不用太过纠结。只要记住一个关键点就好，静态作用域在编译代码，甚至写下代码时就确定好了。

JS的作用域有哪些

如今的JS，作用域包括全局作用域，即当JS代码开始执行时的最初的作用域；函数形成的作用域，即每声明一个函数，函数内部都将形成一个作用域（eval函数会形成特殊的作用域，本文暂不讨论）；以及ES6发布之后的块级作用域，关于块级作用域的详细说明可以查看ES6学习笔记-变量声明和块级作用域。下面举个栗子来说说全局作用域和函数作用域。

执行上下文和变量对象

前面提到的作用域是就一段JS程序代码在JS引擎的编译阶段来说的，编译完成之后，引擎就要执行代码了。这时就要引入另外两个重要概念，执行上下文和变量对象。

先说回前面的代码，对于有过其他编译型语言使用经验的同学或许会有些疑惑，因为这样明显错误的代码在编译的时候可能就已经会提示报错了。但是作为解释执行的JS，错误的抛出是在JS执行每一个具体任务时才进行。上面这段代码整体来说其实就只执行了一个任务，声明了一个变量和两个函数，在这个层面上代码并没有任何错误，所以也就不会报错，至于每个函数内部的代码如果有错误，只会在调用函数的时候才抛出，除非你在函数内部写了一堆符号让JS引擎在编译的时候识别就已经出错了。

但是既然前面说到作用域是在编译阶段进行词法分析的时候就确定好了，那么为什么不在编译时就抛出错误，而非要让程序执行到出错时才报错呢？对于这个疑问我个人是这样来理解的。

首先，在JS引擎内部是只有当函数进行调用的时候才会将其推入调用栈（call stack），执行函数内部的代码；此前函数一直是作为函数对象保存在堆内存中的。所以对于不会影响正常编译流程的语法错误，根本就没有机会抛出错误。

其次，如果在编译的时候明确了作用域的话，JS引擎是否就已经知道，bar函数内的b变量是获取不到的呢？这一点我不敢肯定，但作为浏览器来说，为了用户体验，肯定是希望尽可能让更多可以执行的代码能顺利执行，如果因为一个函数内的错误而在编辑脚本时就报错，多少也就相当于让整个页面的JS脚本挂掉，从而导致更多的内容无法加载，功能无法使用，这样的效果恐怕不太适合浏览器这样的产品场景。

执行上下文

前边提到了一个调用栈（call stack）的概念，其实推入栈的就是执行上下文（Execution Context），也有翻译成执行环境。

还是拿前面例子来说，这段代码作为一个JS文件，浏览器运行的时候，会首先创建全局上下文，push到调用栈的底部，当关闭浏览器窗口后，这个全局上下文才会清除。

然后当我们调用bar()的时候，创建了bar的上下文，然后push到调用栈，开始执行bar，此时接着调用了foo()，创建foo的上下文，开始执行foo；foo执行完毕之后，foo的上下文pop出来，然后继续执行bar后续的部分，bar执行完成之后，bar的上下文弹出，此时又回到全局上下文。

以上是关于执行上下文的一个生命周期流程。当执行上下文创建之后会继续创建当前上下文的变量对象以及作用域链，如果是函数上下文，还会this的值。

变量对象

在JS中，所有的变量，无论是值类型还是引用类型，其实都是一个更大的对象的值，这个对象就是变量对象(Variable Object)。变量对象我们无法直接访问到，但在写JS时却几乎无时无刻不在访问它。

我们可以这样来理解，当我们声明一个变量，声明一个函数，其实在JS处理时，是把它们都放到了一个对象里，key就是变量或函数名，value就是变量的值或者函数的引用。

这就牵扯到JS的两大数据类型————值类型和引用类型。我们知道JS里有五种基本数据类型（ES6以后加上Symbol为六种）：String, Number, Boolean, Undefined, Null。这些都是按值来保存在栈内存中的，都是值类型，又叫原始类型。而Object类型(包括Array, Function)则是按引用保存在堆内存中，这是引用类型。

在变量对象中，值类型数据的变量的自然存储的就是变量真实的值；而引用类型的数据存储的则是堆内存中真实值的引用地址。所以对于初学者会遇到一个经典的坑

会发现b复制了一份a之后，修改b同时也会修改到a，这其实就是因为变量a和b实际上都是同一个对象，即复制a只是复制了在变量对象中a的引用地址到b，a和b实际都指向堆内存中同一个对象。由此，也就带来另一个经典的问题，对象的深浅拷贝。有兴趣的读者不妨自己思考实现一下，如何复制一个对象而不是只复制了一个引用。

关于变量对象还有一点值得说的就是变量的声明提升(Hositing)。

其实就是在用var关键字声明变量的时候，会将变量声明提升到当前作用域的顶部首先进行声明，然后才进行赋值。所以var声明的变量甚至可以在它声明之前的地方进行使用而不会报错，只是此时变量的值是undefined。这种不太符合逻辑的设计其实是JS的历史原因造成的，使用ES6的let会更加严谨。

而要解释所谓的变量声明提升，其实就是在创建变量对象之后，已经可以通过变量对象访问到实际变量了，只是此时由于并未赋值，所以它的值是undefined。

作用域链

当我们调用bar的时候，虽然bar又调用了foo，但是因为JS是静态而非动态作用域，所以调用foo内的变量a和b的作用域规则依然是从声明函数的地方来分析，所以哪怕bar声明了变量a，foo输出的仍然是全局的a = 1。

这其实就是所谓的作用域链，在调用一个函数的时候，变量的查找会首先在本级作用域内进行，如果本级作用域没有找到声明的变量，则沿着本级作用域向上查找，直到顶端的全局作用域，哪一个作用域首先找到了变量声明就停止查找并使用这一个作用域下的变量值，如果没有则会报xxx is not defined这类错误。
